## Rules for Designing Test Cases

### Rule 1: Steps Must Be Natural Language — NEVER Cypress Code
Steps describe WHAT the user does in plain English. They must NEVER contain code, selectors, or Cypress commands.

WRONG (Cypress code):
  "cy.get('#auth-btns .btn-fill').click()"
  "cy.get('#s-name').type(testData.users.newUser.name)"

RIGHT (natural language):
  "Click the Sign Up button in the navigation bar"
  "Enter the new user's name in the Full Name field"

WRONG (selector references):
  "#cart-count should contain '1'"
  ".drawer should have class 'open'"

RIGHT (natural language):
  "Cart badge should display 1"
  "Cart drawer should be visible"

This rule is STRICT. Every step and every assertion must read like a human-written test case that a manual QA tester could follow without seeing any code.

### Rule 2: Each Test Is a Complete Independent Journey
Each test case starts with a fresh browser on the homepage. There is NO shared state between tests. If a test needs items in the cart, it must either:
- Add them via UI steps within the test, OR
- Specify stateSeeding to pre-load localStorage before the page loads

### Rule 3: Every Step Must Be a Concrete Single Action — NO Exceptions
Each step must describe exactly ONE user action or ONE verification. NEVER summarize multiple steps into one. This rule has ZERO exceptions.

WRONG (summary — NEVER do this):
  "Complete the checkout process"
  "Fill in all signup form fields"
  "Go through the entire flow"
  "Perform the checkout steps"

RIGHT (concrete single actions):
  "Click the Choose Delivery button"
  "Select Standard Shipping option"
  "Click Continue to Payment button"

If a test needs a long prerequisite flow (e.g., checkout before testing confirmation), you have TWO options:
  OPTION A: Include ALL prerequisite steps as concrete actions (same as TC-002 does)
  OPTION B: Don't create that test — if TC-002 already covers confirmation assertions, a separate TC-003 is redundant

NEVER use a summary step as a shortcut. If you can't write all the steps, don't create the test case.

### Rule 4: Follow the App Model Flows — Never Skip Steps
For multi-step journeys, use the EXACT flow sequences from the App Model. Do NOT skip steps. Do NOT reorder steps.
- To reach payment: add product → open cart → proceed to checkout → choose delivery → select shipping → continue to payment
- If a button has "enabledWhen" in the app model, the prerequisite action MUST appear as an earlier step
- If a button is disabled by default, the step that enables it MUST come first

### Rule 5: Assert Only on the Current Page
After a navigation trigger (see navigationTriggers in the App Model), you are on a NEW page. Only assert against elements from that page.
- After clicking Place Order: you are on the confirmation page. Assert confirmation elements ONLY.
- NEVER assert cart badge, checkout elements, or homepage elements after navigating away from those pages.

### Rule 6: Use Exact App Message Text in Assertions
For assertion values, use the EXACT strings from appMessages in the App Model. Do NOT paraphrase or guess.
- Signup success: "✓ Account created! Signing you in…" (not "Success" or "Account created")
- Empty fields: "Please fill in all fields." (not "Please fill out this field")
- Invalid email: "Invalid email address." (not "Invalid email format")

### Rule 7: Right Granularity — One User Goal Per Test
Each test = one meaningful user goal with 3-10 interactions.
Good goals: "Add a product and verify cart", "Complete checkout", "Signup with valid details"
Bad goals: "Click a button" (too small), "Do everything in the app" (too big)

### Rule 8: Use State Seeding for Prerequisites That Need Pre-existing Data
When a test requires data that doesn't exist yet (e.g., testing duplicate signup needs an existing account), specify the stateSeeding key in the testData field.
- For duplicate account test: testData should include "stateSeeding": "existingUserForDuplicate"
- For checkout tests that skip the add-to-cart flow: testData should include "stateSeeding": "cartWithOneItem"

### Rule 9: Hidden vs Removed — Use the App Model
When asserting something is gone:
- If the App Model says an element is HIDDEN (display:none, visibility:hidden), assert "should not be visible"
- If the App Model says an element is REMOVED from DOM, assert "should not exist"
- The auth buttons (#auth-btns) are HIDDEN after login/signup, not removed — so assert "not visible"
- The cart drawer is HIDDEN when closed, not removed — so assert "not visible"

### Rule 10: Reference Test Data by Fixture Keys
Use descriptive fixture references, not hardcoded values.
- "the new user's name" (maps to testData.users.newUser.name)
- "the new user's email" (maps to testData.users.newUser.email)
- "valid card number" (maps to testData.payment.validCard.number)
- "the product to add" (maps to testData.products.productToAdd)
